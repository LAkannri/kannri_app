# エンカンAI 機能説明書

> キャリア申請（電気・ガス・ネット）の事務作業を自動化する RPA システムの機能まとめです。
> 対象読者：運用担当者・管理者・開発者。専門用語にはふりがな的な補足を添えています。
> 最終更新：2026-06-25

---

## 1. このアプリは何をするもの？

**エンカンAI** は、SFA（営業管理用の Google スプレッドシート）にたまった申し込み案件を、
各キャリアの **Web 申請フォームへ自動で入力・送信する** ロボット（RPA）です。

- 担当者が一度だけ「お手本」を録画する
- AI（Google Gemini）がその操作から **日本語の手順書** を自動生成する
- 以降はロボット（Playwright）が、その手順書どおりに代理入力する

という思想で作られています。対象ユーザーは **非エンジニアの録画担当者** のため、画面と文言は
徹底的に「やさしく」設計されています（絵文字・かみ砕いた説明）。

---

## 2. 全体像（どの部品が何を担当するか）

| 役割 | 担当技術 | ざっくり説明 |
|---|---|---|
| 顔（画面） | Streamlit（マルチページ） | 担当者が触る操作画面 |
| 脳（設定の保管） | Supabase（`merchants` テーブル） | ロボットの設計図を JSON で保存 |
| 案件データ | Google スプレッドシート（SFA） | 申し込み案件の一覧（顧客情報） |
| 手足（自動操作） | Playwright（Chromium） | 実際にブラウザを操作して入力 |
| AI（手順生成） | Google Gemini（`gemini-2.5-flash`） | 録画から手順書を自動作成 |
| クラウド実行 | GitHub Actions | 担当者の PC を開かずに毎日自動実行 |

```
[Googleスプレッドシート(案件)] --読み取り--> [robot.py] --操作--> [キャリアの申請フォーム]
                                              ^
                              [Supabase: ロボットの設計図・処理済み記録]
```

---

## 3. 登場人物（利用者と役割）

- **録画担当者**：新しいキャリアのフォームを一度だけ録画し、ロボットを作成する人。
- **運用担当者**：日々の案件が正しく処理されているかを見守る人。
- **管理者**：接続キーやクラウド実行（Actions）を設定する人。

## 4. 画面の機能（Streamlit マルチページ）

トップページ（`app.py`）はサマリーのモック表示です。実務の中核は
**「📝 エントリー業務自動化」** ページ（`pages/2_…`）で、4つのビューを切り替えて動きます。

### 4-1. ダッシュボード（`dashboard`）
- 登録済みロボットの一覧を表示。
- 各ロボットの **稼働中／停止** を切り替え。
- 「新しく作る」→ 基本情報入力へ。「設定を開く」→ 司令室へ。

### 4-2. 基本情報（`step1_basic`）
- ロボット名、商材種別（ネット／電気／ガス／その他）、SFA スプレッドシートの URL などを設定。

### 4-3. 録画（`step2_record`）
- Playwright の **codegen**（録画機能）でお手本操作を記録。
- 記録した操作を **Gemini** に渡し、日本語の手順書（steps）を自動生成。

### 4-4. 司令室（`project_room`）
- 自動生成された手順書を **人が確認・修正** できる。
- **条件分岐ルール** の作成・編集。
- **お試し実行**（モック顧客データでロボットを単体テスト）。

> 補足：準備中のページ（全状況進捗確認／開通進捗反映／変更キャンセル／その他設定）は
> プレースホルダで、今後の拡張枠です。

## 5. ロボットエンジンの仕組み（`robot.py`）

ロボットの挙動は、すべて Supabase の設計図（`config_json`）に従います。

### 5-1. 手順書（steps）
各ステップは「順番・いつ・操作・対象・値・変換・ai_code」を持ちます。
日本語キー（旧）と英語キー（新）の両方に対応しています。

```json
{ "順番": 1, "いつ": "常に", "操作": "文字を入力", "対象": "お名前",
  "値": "{顧客_氏名}", "変換": "", "ai_code": "..." }
```

- **操作のマッピング**：`文字を入力→fill` / `クリック→click` / `選択→select` / `チェック→check`
- **動的置換**：`値` や `ai_code` の中の `{項目名}` は、実行時にスプレッドシートの該当列の値に
  置き換わります（例：`{電話番号}` → `090-1234-5678`）。`090` が数値化して頭の 0 が消えないよう、
  **純粋な文字列置換** で行います。

### 5-2. 値の変換（コード不要の加工）
`変換` を指定すると、置換後の値を加工できます（`apply_transform`）。

| 変換 | 例：`090-1234-5678` → |
|---|---|
| ハイフン除去 | `09012345678` |
| 数字のみ | `09012345678` |
| 市外局番 | `090` |
| 市内局番 | `1234` |
| 加入者番号 | `5678` |
| 郵便番号_上3桁 / 下4桁 | `814` / `0165`（`814-0165` の場合） |

### 5-3. 条件分岐ルールエンジン（`conditions`）
ハードコードせず、設定で分岐を表現します（`evaluate_condition`）。
各ステップの `いつ` にルール名を指定すると、その条件が成立したときだけ実行されます。

```json
{ "name": "未成年ルート", "logic": "AND",
  "rules": [ { "col": "年齢", "op": "lt", "value": "20" } ] }
```

- **演算子（op）**：`eq`(一致) / `ne`(不一致) / `contains`(含む) / `not_contains`(含まない) /
  `empty`(空) / `not_empty`(空でない) / `gt`(より大) / `gte`(以上) / `lt`(より小) / `lte`(以下) /
  `in`(カンマ区切りのいずれか)
- **logic**：`AND`（全条件）／`OR`（いずれか）
- **安全側の既定**：`常に`・空 → 必ず実行。**未定義のルール名 → スキップ（False）** にして事故を防止。

### 5-4. 入力の二段構え（AI呪文 → 汎用フォールバック）
1. **AIの「最強の呪文」（`ai_code`）**：サイト固有のコードをまず実行（`exec`）。
2. 失敗したら **汎用フォールバック**：プレースホルダ／ラベル／テキスト／ロールなど、
   複数の探し方で要素を見つけて操作。

「送信・確認・次へ」を含むボタンは Submit ボタンも候補に加えるなど、フォーム送信を取りこぼしにくくしています。

### 5-5. 安全装置（申請漏れ・誤申請の防止）
- **入力漏れで即停止**：対象が見つからない等の重大エラーが起きたら、以降の **送信を実行しない**
  （`has_critical_error` フラグ）。中途半端な申請を防ぎます。
- **ボット検知で安全停止**（`_looks_blocked`）：CAPTCHA／reCAPTCHA／Turnstile 等を検出したら、
  送信せずに中止。
- **証跡スクショ**（`_save_screenshot`）：失敗・中止・CAPTCHA 時に `artifacts/` に画面を保存。
  クラウドでは目視できないため重要な記録になります。
- **目視待機**：有人（ローカル）実行では結果を確認できるよう操作をゆっくり行い、終了前に 10 秒待機。

## 6. データ連携

### 6-1. SFA スプレッドシート（リンク共有・読み取り専用）
`fetch_pending_rows` がスプレッドシートを **CSV として読み取り**、未エントリー行を取得します。

- **接続方式**：認証なし。スプシの共有を「**リンクを知っている全員（閲覧者）**」にする。
  gviz エンドポイント `…/gviz/tq?tqx=out:csv&sheet=<タブ名>`（`_csv_export_url`）で取得。
  非公開だとログイン画面（HTML）が返るため、分かりやすいエラーで知らせます。
- **列のマッピング**：スプシの **ヘッダ名がそのまま** `{項目名}` に対応します（列『電話番号』→ `{電話番号}`）。
- **絞り込み**：`trigger_col`（既定『ステータス』）が `trigger_val`（既定『未エントリー』）の行のみ対象。

### 6-2. Supabase（設計図と処理済み記録）
- `merchants` テーブルにロボットごとの設計図（`config_json`）を保存。
- **処理済み記録**（`config_json._processed_keys`）：後述の二重申請防止に使用。

## 7. クラウド自動実行（GitHub Actions）

担当者の PC を開かなくても、クラウドでロボットを動かせます
（`.github/workflows/run-robots.yml`）。

- **スケジュール**：毎日 **JST 08:00（UTC 23:00）** に `python robot.py --all` を自動実行。
- **手動実行**：Actions タブの「Run workflow」からも起動可能。
- **実行モード**：
  - `--all`＝稼働中（`is_active=True`）の全ロボットを処理（`run_all_active`）。
  - 引数にロボット名 → そのロボットをモック顧客で単体実行。
- **headless 切替**（`is_headless`）：`ENKAN_HEADLESS=1/0` を明示。未指定なら `CI` 環境で自動 headless。
- **二重起動防止**：`concurrency` で同時実行を抑止（二重申請の防止）。
- **証跡アップロード**：`artifacts/` を成果物として保存（14日保持）。

### 7-1. 本番ゲート（誤申請の最重要対策）
- 環境変数 **`ENKAN_ALLOW_LIVE`** が `1` のときだけ実申請。
- 既定は **ドライラン**（対象件数と内容を表示するだけ／実ブラウザ操作なし）。
- ワークフローでは **手動実行で `live=true` のときだけ ON**、
  **スケジュール実行は常にドライラン**にしてあり、事故を防ぎます。

### 7-2. 二重申請の防止（書き戻しできない代わりの仕組み）
読み取り専用のためスプシのステータスを書き戻せません。そこで、

- 処理済み行のハッシュ（`_row_key`、**ステータス列は除外**）を
  **Supabase の `config_json._processed_keys`** に保存。
- 再実行時は処理済みの行を **自動スキップ**。
- 肥大化を防ぐため **直近 5000 件** に制限。

## 8. 接続キー（secrets）

`.streamlit/secrets.toml`（ローカル）または環境変数（クラウド）で設定します。
**機密情報なのでコミット禁止**（`.gitignore` 済み）。

```toml
SUPABASE_URL    = "https://xxxxx.supabase.co"
SUPABASE_KEY    = "eyJhbGc..."   # anon key
GEMINI_API_KEY  = "AIzaSy..."
```

- ローカル：`.streamlit/secrets.toml`
- クラウド：リポジトリの Settings → Secrets and variables → Actions に 3 つを登録
  （`robot.py:load_secrets` が環境変数を優先）。

## 9. セキュリティ・既知の制約

- **`exec()` による動的実行**：`ai_code` を `exec` で実行しています。AI 生成コードを動かす設計のため、
  **信頼できる入力が前提**。将来は構造化アクションへの置き換えを予定（ロードマップ）。
- **ステータス書き戻し不可**：読み取り専用のため、スプシの「ステータス」列は自動更新されません
  （担当者の目視では未処理のまま見える）。二重申請は `_processed_keys` で防止。
  書き戻しが必要ならサービスアカウント方式への切替が前提です。
- **空セルの正規化**：手順書編集で空セルが `NaN`/`None` になる問題に対し、保存時に空文字へ正規化します。

## 11. 用語集

| 用語 | 意味 |
|---|---|
| RPA | 人手の PC 操作をソフトで自動化する仕組み |
| SFA | 営業管理。ここでは案件一覧の Google スプレッドシート |
| 手順書（steps） | ロボットが従う操作の一覧（日本語で編集可能） |
| 条件分岐（conditions） | 「○○のときだけ実行」を設定で表現する仕組み |
| 司令室 | 手順の確認・編集・テストを行う画面 |
| ドライラン | 実際の申請をせず、対象を表示するだけの安全モード |
| headless | 画面を表示せずにブラウザを動かすモード（クラウド向け） |
| 証跡（artifacts） | 失敗時などに残すスクリーンショット等の記録 |

### 単体テストの実行例

```bash
# 指定ロボットをモック顧客で単体実行（司令室の「お試し実行」相当）
python robot.py "<ロボット名/プロジェクトID>"

# 稼働中の全ロボットをまとめて実行（既定はドライラン）
python robot.py --all
```
